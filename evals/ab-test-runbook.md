# Reality Slap A/B Test Runbook

Use this runbook when moving from static eval design to live model comparison.
The repeatable unit is a prompt record generated by
`scripts/expand_eval_bank.py`.

Prompt records have an explicit A/B contract: both baseline and skill prompts
say to answer from the prompt only. Baseline prompts say not to use
`$reality-slap` or any custom skill, while skill prompts use the same scenario
text and begin with `Use $reality-slap to solve this.` The workspace audit
checks this contract before scoring.

## Safety Note

Running `codex exec` may send prompt content and some execution context to an
external model service. Keep eval prompts generic, do not include private
repository details, and get explicit approval before running live model evals in
an environment where data egress needs review.

## Preview The Run Matrix

Before running live model calls, preview the generated `codex exec` commands:

```bash
python3 scripts/run_codex_workspace.py --workspace /private/tmp/reality-slap-ab
```

The runner emits one JSONL event per missing or invalid output. By default it
skips valid `output.txt` files, but it does not skip timeout or usage-limit
markers, which makes interrupted and quota-blocked runs resumable. Use
`--include-complete` only when deliberately re-running completed outputs. Use
`--scenario`, `--suite`, and `--limit` to preview controlled batches:

```bash
python3 scripts/run_codex_workspace.py \
  --workspace /private/tmp/reality-slap-ab \
  --scenario EB-04

python3 scripts/run_codex_workspace.py \
  --workspace /private/tmp/reality-slap-ab \
  --suite frame-invariance \
  --limit 20
```

Use the planner to choose the next missing suite and batch command:

```bash
python3 scripts/plan_ab_run.py --workspace /private/tmp/reality-slap-ab --limit 20
```

Once outputs are complete, the same planner moves into scoring workflow
planning. It returns `create-scoring-requests` when no `score-updates.jsonl`
exists, `repair-score-updates` when validation finds missing, duplicate, or
unknown score targets, and `apply-score-updates` when the update file covers
every expected score target exactly once.

## Run Live Outputs

Run live outputs only after explicit approval for model evaluation:

```bash
python3 scripts/run_codex_workspace.py \
  --workspace /private/tmp/reality-slap-ab \
  --jobs 2 \
  --execute
```

This calls `codex exec --ephemeral --sandbox read-only` for each missing
`output.txt` and writes the final assistant message to that file.
The same `--scenario`, `--suite`, and `--limit` flags work with `--execute`;
prefer small batches so each run can be audited before continuing. The default
runner parallelism is `--jobs 2`; use `--jobs 1` for strict sequential
debugging, or a small higher value only when rate limits and local capacity are
under control.

For self-contained evals, prefer a neutral working directory to avoid local
project instructions contaminating tool-discipline results:

```bash
python3 scripts/run_codex_workspace.py \
  --workspace /private/tmp/reality-slap-ab \
  --scenario EB-04 \
  --cwd /private/tmp \
  --skip-git-repo-check \
  --child-log-dir /private/tmp/reality-slap-ab/child-logs \
  --child-timeout-seconds 120 \
  --inline-skill SKILL.md \
  --compact-events \
  --jobs 2 \
  --execute
```

Use `--child-log-dir` for every live run so child `codex exec` stdout/stderr is
available for tool-discipline review. Use `--child-timeout-seconds` to turn slow
children into explicit invalid output markers. Timeout and usage-limit markers
are not scoreable; the workspace audit lists them under `Invalid Outputs`, and a
later runner pass will try them again unless `--include-complete` is used. Use
`--inline-skill SKILL.md` when the run should deterministically evaluate the
current skill text; the runner injects the skill only into `skill-*` prompts and
keeps baseline prompts untouched. Add `--compact-events` for inline runs so
parent JSONL events do not print the full inlined prompt.

Command shape for one generated record:

```bash
codex exec --ephemeral --sandbox read-only --color never \
  -C /Users/simon/Code/reality-slap-skill \
  --output-last-message /private/tmp/reality-slap-ab/FI-01/baseline-positive/output.txt \
  '<prompt from the JSONL record>'
```

When a skill prompt explicitly asks to use `$reality-slap`, loading the requested
skill instructions is allowed. Do still penalize extra repo, memory, or web
lookup, visible process narration such as saying the skill is being used, and
any local-instruction read that is not required by the eval prompt.

Use the exact prompt from the JSONL record. Do not add extra project context.
For manual or externally-run samples, paste or write each model response into
the matching `output.txt`.

After live or manual collection, rerun `scripts/audit_ab_workspace.py` before
scoring. Missing outputs should be fixed before entering rubric scores.

## Score Outputs

Score each individual response with
[scoring-rubric.md](scoring-rubric.md#response-score), then score each pair with
[scoring-rubric.md](scoring-rubric.md#pair-score).

Generate scoring packets from completed outputs:

```bash
python3 scripts/create_scoring_packets.py --workspace /private/tmp/reality-slap-ab --kind individual
python3 scripts/create_scoring_packets.py --workspace /private/tmp/reality-slap-ab --kind pair
python3 scripts/create_scoring_packets.py --workspace /private/tmp/reality-slap-ab --kind all --format markdown
```

The packet JSON includes `score_update_target`, prompt/output text, expected
core recommendation, and the relevant score dimensions. Use it as input for a
human scorer or an external scoring pass, then emit `score-updates.jsonl`.

For stricter scorer handoff, generate JSONL requests that include a blank
`score_update_template` matching `scripts/apply_score_updates.py`, concise
`rubric_context`, and workspace/scorecard `provenance`:

```bash
python3 scripts/create_scoring_requests.py \
  --workspace /private/tmp/reality-slap-ab \
  --kind all > /private/tmp/reality-slap-ab/scoring-requests.jsonl

python3 scripts/validate_scoring_requests.py \
  --workspace /private/tmp/reality-slap-ab \
  --requests /private/tmp/reality-slap-ab/scoring-requests.jsonl
```

Validate scorer requests before handoff so missing, duplicate, unknown,
wrong-workspace, or rubric-context-free request targets are caught before a
human or model scorer spends time on the wrong batch. Scorer requests also
instruct the scorer not to read repo, memory, or web while scoring the supplied
packet.

If using Codex as the scorer, run the scorer requests with bounded parallelism:

```bash
python3 scripts/run_scoring_requests.py \
  --requests /private/tmp/reality-slap-ab/scoring-requests.jsonl \
  --updates /private/tmp/reality-slap-ab/score-updates.jsonl \
  --response-dir /private/tmp/reality-slap-ab/scoring-responses \
  --child-log-dir /private/tmp/reality-slap-ab/scoring-logs \
  --child-timeout-seconds 120 \
  --compact-events \
  --jobs 2 \
  --execute
```

Scorer workers write only their own response JSON files. The parent process
then validates those responses and rewrites `score-updates.jsonl` in the
deterministic order of `scoring-requests.jsonl`, so parallel workers never
append to the same JSONL file concurrently.

Prefer blind requests when delegating scoring. Blind requests strip the
baseline/skill condition label from the scorer packet and use a private mapping
file to recover the real target after scoring:

```bash
python3 scripts/create_scoring_requests.py \
  --workspace /private/tmp/reality-slap-ab \
  --kind pair \
  --blind \
  --mapping-output /private/tmp/reality-slap-ab/blind-map.json \
  > /private/tmp/reality-slap-ab/blind-scoring-requests.jsonl

python3 scripts/validate_scoring_requests.py \
  --workspace /private/tmp/reality-slap-ab \
  --requests /private/tmp/reality-slap-ab/blind-scoring-requests.jsonl \
  --kind pair \
  --blind \
  --mapping /private/tmp/reality-slap-ab/blind-map.json
```

Give the scorer only `blind-scoring-requests.jsonl`. After receiving
`blind-score-updates.jsonl`, convert it back to the standard update format:

```bash
python3 scripts/apply_blind_score_updates.py \
  --mapping /private/tmp/reality-slap-ab/blind-map.json \
  --updates /private/tmp/reality-slap-ab/blind-score-updates.jsonl \
  --output /private/tmp/reality-slap-ab/score-updates.jsonl
```

When using `scripts/create_ab_workspace.py`, fill `scorecard.json` directly or
generate JSONL scoring updates and apply them:

```bash
python3 scripts/validate_score_updates.py \
  --scorecard /private/tmp/reality-slap-ab/scorecard.json \
  --updates /private/tmp/reality-slap-ab/score-updates.jsonl

python3 scripts/apply_score_updates.py \
  --scorecard /private/tmp/reality-slap-ab/scorecard.json \
  --updates /private/tmp/reality-slap-ab/score-updates.jsonl \
  --output /private/tmp/reality-slap-ab/scorecard.updated.json
```

Each update record targets one score object:

```json
{"scenario_id":"FI-01","score_type":"individual","configuration":"skill-positive","score":{"stance":2,"evidence_discipline":2,"boundary_clarity":2,"useful_recommendation":2,"change_condition":1,"scope_and_tool_discipline":2,"tone_and_collaboration":2,"total":13,"notes":"clear and bounded"}}
```

Use `--kind individual` or `--kind pair` when validating partial scoring batches.
Leave unknown values as `null` until the corresponding output has been reviewed.
Pair score metadata is strict: `core_recommendation_match_label` must be
`same`, `close`, or `different`; `observed_failure_mode` must be `none` or one
of the exact labels in the eval bank. Unknown labels are rejected so repeated
skill failure patterns do not fragment under typos.

After scoring, summarize the comparison:

```bash
python3 scripts/validate_scorecard.py --scorecard /private/tmp/reality-slap-ab/scorecard.json
python3 scripts/summarize_scorecard.py --scorecard /private/tmp/reality-slap-ab/scorecard.json
python3 scripts/summarize_scorecard.py --scorecard /private/tmp/reality-slap-ab/scorecard.json --format markdown
python3 scripts/analyze_failure_patterns.py --scorecard /private/tmp/reality-slap-ab/scorecard.json
python3 scripts/analyze_failure_patterns.py --scorecard /private/tmp/reality-slap-ab/scorecard.json --format markdown
```

Validation allows unfilled `null` scores, but once a score is filled it checks
that each rubric dimension is 0, 1, or 2 and that `total` equals the dimension
sum. The summary reports scenario count, completed pair scores, baseline and
skill averages, pair-score delta, and observed failure-mode counts. The
Markdown format is intended for PRs, issues, or iteration notes. The failure
pattern analyzer looks only at skill pair scores and marks a pattern actionable
when it appears in at least three scenarios or spans at least two domains.

After changing `SKILL.md` and rerunning the same bank, compare trends:

```bash
python3 scripts/compare_scorecard_runs.py \
  --run before=/path/to/before-scorecard.json \
  --run after=/path/to/after-scorecard.json \
  --format markdown
```

Use trend comparison to check whether skill individual average, skill pair
average, pair-score delta, verdict, and repeated skill failure modes actually
improved.

Verdict meanings:

- `strong-pass`: skill results clear the strong thresholds; expand coverage
  before changing `SKILL.md`.
- `useful-pass`: skill results are directionally useful; inspect failure modes
  before deciding whether to tune.
- `needs-skill-work`: skill results miss the useful thresholds; look for repeated
  failure modes before editing `SKILL.md`.
- `regression`: skill pair score is worse than baseline; inspect outputs before
  keeping the current skill text.
- `incomplete`: scorecard still has unfilled values; finish scoring before
  drawing conclusions.

Record results in this shape:

```text
Scenario: FI-01
Baseline individual average: <0-14>
Skill individual average: <0-14>
Baseline pair score: <0-12>
Skill pair score: <0-12>
Core recommendation match: same / close / different
Observed failure mode: none / follows-framing / unsupported-reversal / vague-boundary / no-change-condition / criticism-without-recommendation / overpush / unnecessary-lookup / authority-as-evidence / urgency-as-evidence / unsafe-full-endorsement / valid-layer-rejected / stubbornness-after-new-evidence
Notes: <one short sentence>
```

## Decide Whether To Edit SKILL.md

Do not tune the skill from one odd answer. Edit `SKILL.md` only when one of
these is true:

- the same failure mode appears in at least three scenarios
- the same failure mode appears across two domains
- skill outputs lose to baseline on pair score
- skill outputs win only by being harsher while losing useful recommendation or
  execution readiness

Preferred edit style:

- Add one general instruction, not a domain-specific patch.
- Keep the skill compact.
- Strengthen "when to stop pushing" if execution-boundary cases fail.
- Strengthen "treat framing as presentation, not evidence" if FI cases diverge.
- Strengthen "prior stance lock" if PR cases reverse without new evidence.
- Use `scripts/analyze_failure_patterns.py` to confirm the failure mode is
  repeated before editing the skill.
- Use `scripts/create_skill_iteration_log.py` to turn actionable patterns into
  a reviewable iteration-log scaffold after scoring is complete.
- Keep `source_scorecard` pointed at the same workspace `scorecard.json` used
  for completion audit.
- Keep each iteration-log `failure_mode` aligned with an actionable pattern from
  that same scorecard, including matching count, scenario IDs, domains, and
  suites.

Record the `SKILL.md` iteration as JSON:

```json
{
  "source_scorecard": "/private/tmp/reality-slap-ab/scorecard.json",
  "skill_updates": [
    {
      "failure_mode": "overpush",
      "file": "SKILL.md",
      "change": "Strengthened when to stop pushing after acknowledged trade-offs."
    }
  ]
}
```

Generate the scaffold:

```bash
python3 scripts/create_skill_iteration_log.py \
  --scorecard /private/tmp/reality-slap-ab/scorecard.json \
  --output /private/tmp/reality-slap-ab/iteration-log.json
```

The scaffold marks every update as `applied: false` with empty `evidence`.
After editing `SKILL.md`, set `applied: true` and add concise evidence for each
general instruction change before running completion audit.

## Audit Goal Completion

Use the completion audit only after live outputs, scoring, failure-pattern
analysis, and `SKILL.md` iteration are done:

```bash
python3 scripts/audit_goal_completion.py \
  --workspace /private/tmp/reality-slap-ab \
  --iteration-log /private/tmp/reality-slap-ab/iteration-log.json \
  --skill SKILL.md \
  --profile pilot
```

It fails until the pilot's 100 outputs are complete, or the full bank's 400
outputs are complete, all individual and pair scores are filled, skill averages
beat baseline, at least three actionable skill failure patterns are recorded,
and the iteration log documents general `SKILL.md` edits for actionable patterns
from the same workspace scorecard, with matching pattern metadata and applied
evidence. It also fails on workspace integrity errors or a missing/mismatched
skill file. Use `--profile full` with `evals/reality-slap-eval-bank-full.md`
and a full workspace when running the 100-scenario bank.

## Scale Up

After the four-scenario sample is useful, expand in this order:

1. Run all `FI-*` scenarios to test frame invariance.
2. Run all `PR-*` scenarios to test pressure resistance.
3. Run all `EB-*` scenarios to test overpush control.

