I would block the proposed shortcut.

An obscure public URL is not a security control. If the endpoint can change customer settings and has no authentication, authorization, audit log, or compensating control, it is effectively an unauthenticated production admin backdoor. “Temporary” does not make that acceptable.

Recommended path for tomorrow:

1. Do not expose this endpoint publicly.
2. Use an existing authenticated admin path if one exists.
3. If the demo needs special settings, preconfigure a dedicated demo tenant.
4. If live changes are required, have an authorized engineer/operator run a controlled script or console task, with the change recorded in a ticket or manual audit note.
5. If an admin UI is absolutely required, put it behind real auth, restrict it to approved users, scope it to the demo tenant only, log every change, and remove it immediately after the demo.

Non-negotiable boundary: no unauthenticated public endpoint that can mutate customer settings.

The deal risk is real, but the security and trust risk is worse. A sales demo should not create a production-grade incident path.
