Block the proposed endpoint, but do not block the demo.

My recommendation: ship a controlled tomorrow-ready admin path instead:

1. Use a demo-only tenant or staging environment if possible.
2. Run the setting change through an internal script, console command, or existing authenticated admin surface.
3. If an HTTP endpoint is unavoidable, put it behind existing auth or VPN, restrict it to named users/IPs, allow only the exact demo tenant and fields, log every invocation, and set an automatic removal/disable time.
4. Get explicit owner sign-off if real customer settings can be changed.

The non-negotiable boundary is: no public unauthenticated endpoint that mutates customer settings.

An obscure URL is not a security control. It can leak through chat, browser history, logs, referrers, screenshots, analytics, or accidental discovery. Because this endpoint changes customer settings and has no audit trail, a mistake or abuse would be hard to detect, attribute, or unwind.

So the practical answer is not “be rigid and miss the demo.” It is: protect the demo by narrowing the workflow to a controlled, auditable, temporary path.
