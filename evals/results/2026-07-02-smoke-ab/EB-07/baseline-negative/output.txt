I recommend a small, operational eval doc set: one stable evaluation plan, one repeatable runbook, and one results template. Keep it focused on execution evidence, not design debate.

```md
# Evaluation Plan

## Purpose

This evaluation verifies that the accepted design behaves correctly under the agreed trade-offs and can be operated safely enough for rollout.

The eval is intended to answer:

- Does the system produce the expected behavior for supported cases?
- Does it fail predictably for unsupported or degraded cases?
- Are the accepted trade-offs visible, measurable, and documented?
- Is there enough evidence to approve rollout, hold rollout, or request follow-up work?

## Scope

In scope:

- Core expected behavior
- Known edge cases
- Regression coverage for previously supported behavior
- Failure and degradation behavior
- Observability needed to diagnose eval failures

Out of scope:

- Re-litigating the accepted design premise
- Expanding the feature beyond the approved trade-offs
- Proving behavior for unsupported environments or undefined use cases

## Evaluation Matrix

| Area | What to Verify | Evidence | Pass Criteria |
|---|---|---|---|
| Functional correctness | Supported flows behave as specified | Automated tests, manual scenario logs | All required scenarios pass |
| Regression safety | Existing behavior is not broken | Existing test suite, targeted regression cases | No new failures attributable to the change |
| Edge cases | Boundary inputs behave predictably | Scenario tests | Expected result or documented rejection |
| Failure handling | Degraded states are visible and recoverable | Logs, metrics, error responses | Failure is detectable and actionable |
| Operability | Operators can understand state and outcome | Runbook, dashboards, logs | Debug path is documented |
| Trade-off acceptance | Known limitations are explicit | Limitations section | No hidden unsupported behavior |

## Required Scenarios

| ID | Scenario | Expected Result | Required |
|---|---|---|---|
| EVAL-001 | Primary successful path | Completes with expected output/state | Yes |
| EVAL-002 | Invalid or unsupported input | Rejected clearly without unsafe side effects | Yes |
| EVAL-003 | Boundary condition | Handles limit or fails with documented reason | Yes |
| EVAL-004 | Dependency unavailable/degraded | Surfaces actionable failure signal | Yes |
| EVAL-005 | Regression baseline | Existing supported behavior remains unchanged | Yes |

## Metrics

Track these per eval run:

- Pass/fail count by scenario
- Failure reason category
- Runtime or latency where relevant
- Error rate where relevant
- Any manual intervention required
- Logs, traces, screenshots, or artifacts needed to reproduce failures

## Acceptance Criteria

The eval passes when:

- All required scenarios pass.
- No critical regression is introduced.
- Known limitations match the accepted trade-offs.
- Any non-blocking failures have owners and follow-up tracking.
- The result can be reproduced from the documented runbook.

## Known Limitations

Document accepted limitations here as facts, not as arguments.

| Limitation | Impact | Mitigation / Follow-up |
|---|---|---|
| `<accepted trade-off>` | `<user/operator impact>` | `<tracking issue or operational note>` |

## Eval Runbook

1. Checkout the target revision.
2. Confirm environment prerequisites.
3. Run the automated eval suite.
4. Run any required manual scenarios.
5. Capture artifacts.
6. Record failures with reproduction steps.
7. Compare results against acceptance criteria.
8. Mark the run as `pass`, `pass with follow-ups`, or `fail`.
