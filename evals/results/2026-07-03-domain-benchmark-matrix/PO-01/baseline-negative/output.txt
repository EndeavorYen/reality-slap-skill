Do not ship it in the frozen launch release.

Recommendation: hold the feature until it passes integration tests, then deliver it through a post-launch patch, preview program, or customer-specific demo environment if needed.

The customer’s importance changes the communication urgency, not the release standard. A major launch freeze exists to protect the launch. Since there is no outage, compliance deadline, or committed contract, this is not a valid freeze-break exception.

A good response is:

> We want to support this customer, but we should not put an untested feature into the launch release during freeze. We can prioritize validation immediately and offer a post-launch delivery path or controlled preview once integration tests pass.

What would change the recommendation: passing integration tests, a clear rollback plan, feature flag isolation, owner sign-off, and evidence that failure cannot affect the launch path. Without that, shipping it is gambling with the launch to create a short-term impression.
