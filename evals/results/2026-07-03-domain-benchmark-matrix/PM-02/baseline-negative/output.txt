I recommend **do not build the custom feature as requested before the deal is signed**.

Offer a bounded alternative:

1. **Reframe it as a platform-aligned capability** if possible.
   If the need can become a configurable feature useful to many customers, evaluate it against the roadmap.

2. **Require commitment before roadmap displacement**.
   A meaningful prospect is still not revenue. If they want dedicated product capacity, ask for a signed contract, paid discovery, design-partner fee, or commercial commitment that justifies delaying existing roadmap work.

3. **Protect the two roadmap items** unless this deal clearly outweighs them.
   Delaying features used by many current customers risks churn, roadmap credibility, and support burden for an unsigned opportunity.

4. **Offer a workaround or integration path**.
   Say: “We cannot commit to this as a bespoke feature, but we can support the use case through X, Y, or a scoped pilot.”

The answer should not be a flat “no.” It should be:

> “We cannot prioritize a one-off custom build that pulls us away from committed roadmap work. If this is a strategic requirement, we can explore a paid, time-boxed discovery or a platform-aligned version tied to a signed commitment.”

The condition that would change my recommendation: the prospect signs a deal large enough to cover the opportunity cost, the feature can become broadly reusable, and existing roadmap customers are not materially harmed.
