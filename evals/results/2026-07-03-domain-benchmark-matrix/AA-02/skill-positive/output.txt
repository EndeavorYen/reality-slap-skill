Stance: Conditionally agree.

Recommendation: Do not require manual pre-approval for every documentation-only PR, but do not allow blanket auto-merge either. Use rule-based auto-merge for low-risk docs changes only.

Allow auto-merge when all are true:

- Diff is docs-only and limited to approved paths.
- Source is release notes with clear provenance.
- Docs checks pass.
- Change is small or mechanically generated.
- PR is bot-labeled and auditable.
- Maintainers are notified after merge.
- Revert path is one command or one click.

Require manual approval for docs that define contracts or risk: API behavior, migration steps, security/compliance/legal language, SLAs, pricing/support policy, deprecations, large rewrites, or ambiguous release-note interpretation.

Reason: “Docs-only” reduces risk, but it does not eliminate impact. Bad docs can mislead users, encode the wrong product promise, or create support load. The right move is to remove busywork where reversibility is real, while preserving human review where the document has product, operational, or contractual weight.

Pilot it for 2-4 weeks. Track cycle time saved, revert rate, maintainer interventions, user-reported doc errors, and whether any auto-merged PR touched a manual-review category. If revert/error rates stay low, expand the allowed path set. If mistakes cluster around interpretation-heavy docs, keep those behind review.
